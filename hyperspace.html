Hyperspace Defender

This game was made for the University of Colorado Game Development Specialization. The task was to create a 2D shooter inspired by the arcade space games of the early 80’s. We were to concentrate mainly on the game mechanics, including movement, object collision, and enemy behavior. I went a few steps further and made original 3D models in Blender, and an original song and sound effects in Ableton Live. You can play by visiting the itch.io link below.

The game is one endless level, where the player tries to defend against an endless wave of galactic threats. While the gameplay is derivative of some of the greats like Galaga, Space Invaders, and Gradius, I used some features that helped optimize the speed and development of the game.


Object Pooling

I utilized object pooling to make use of resources more efficient. Pooling allows the engine to pre-load many objects that would have to be instantiated very often and setting the active state to false, ready to activate whenever it is needed. Object pooling was used for player and enemy bullets, visual effects (explosions), and item pickups.

For example, when a player shoots, a bullet would normally have to be instantiated at the ship’s position. When the bullet hits an enemy or exits the screen, that object would need to be destroyed. This can be costly if many bullets are constantly entering and leaving the scene.

With object pooling, a large amount of bullets are instantiated at the start of the level. When the player shoots, the system instead places an inactive bullet from the pool at the position of the ship, and sets it to active. The bullet is then set to inactive and returned to the pool instead of being destroyed. If there are no available inactive bullets, the pooling system instantiates a set number of extra bullets and places them in the inactive pool.

Enemy Behavior Inheritance

I was able to use inheritance for enemy behavior, saving development time on the creation of new enemy types. While each alien color and the UFO move and attack differently, they all share a Parent Class for base enemy behavior. Each Child Class can then implement new behaviors such as movement variations, life-points, or frequency of attack. See below for how each enemy differs.
